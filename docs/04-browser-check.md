# 04. ブラウザで確認して改善する

ここでは、生成されたアプリを実際にブラウザで見て、Copilotに改善してもらいます。

コード生成だけで終わらせず、画面を見ながら小さく直すことがこの章の目的です。

## この章でやること

- VS Code内ブラウザまたはPortsプレビューで画面を見る
- スマートフォン幅で主要フローを確認する
- `/web-design-reviewer` で見た目の問題を直す
- `/webapp-testing` で主要フローの動作を確認する

## 開発サーバーを起動する

まだ起動していない場合は、`app/` で開発サーバーを起動します。

```bash
cd app
npm run dev
```

Codespacesでは、Portsパネルの5173番を開きます。ローカルVS Codeでは、ターミナルに表示されたURLを開きます。

## まず人間が見る

最初に、自分たちで主要フローを1回触ってみます。

- 最初の画面で何をするアプリか分かるか
- スマートフォン幅で文字やボタンが見やすいか
- ボタンや入力欄を迷わず操作できるか
- 結果画面や完了画面がデモしやすいか
- LINEミニアプリらしさが伝わるか

ここで見つけた違和感を、短い言葉でメモしておきます。

## 見た目を改善する

VS Code内ブラウザやポートプレビューを見ながら、Awesome由来のレビューSkillを使います。

```text
/web-design-reviewer

スマートフォン幅で画面を確認し、デモに影響する見た目の問題だけ小さく直してください。
```

具体的に気になる点がある場合は、一緒に伝えます。

```text
初回表示で何をするアプリか分かりにくいです。
ヘッダーと説明文を改善してください。
```

```text
結果カードが窮屈なので、余白と文字サイズを調整してください。
```

```text
LINEプロフィールMockをもっと自然に画面へ反映してください。
```

## 主要フローを確認する

見た目を整えたら、Awesome由来のテストSkillで主要フローを確認します。Playwright MCPが使える場合は補助として使います。

```text
/webapp-testing

初期表示から完了画面まで、主要フローを1回通して確認してください。
スマートフォン幅で、クリック、入力、画面遷移、結果表示を見てください。
```

エラーや表示崩れが見つかった場合は、Copilotに修正させます。修正後は、同じ流れをもう一度確認します。

## 直す範囲を絞る

この章では、デモに効く問題だけ直します。

直す対象の例:

- 文字が読みにくい
- ボタンが押しづらい
- 画面から要素がはみ出す
- 主要フローの途中で進めなくなる
- 結果画面が何を伝えているか分かりにくい

発展課題に回す例:

- 本物のLIFF接続
- バックエンド保存
- 複雑なアニメーション
- 画面数を増やす改善
- Playwrightの自動テスト追加

## 次の章へ進む目安

次の状態になったら、成果共有へ進みます。

- 主要フローをブラウザで1回通している
- スマートフォン幅で大きな崩れがない
- 1回以上Copilotに改善してもらっている
- 残りの改善点を発展課題として説明できる
